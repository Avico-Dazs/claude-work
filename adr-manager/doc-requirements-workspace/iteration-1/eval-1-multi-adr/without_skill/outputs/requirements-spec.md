# 要件定義書: EC購入フロー（カートから決済完了まで）

**バージョン:** 1.0
**作成日:** 2026-03-22
**ステータス:** ドラフト

---

## 1. 概要

### 1.1 目的

本書は、ECサイトにおいてユーザーが商品をカートに追加してから決済を完了するまでの一連のフローに関するシステム要件を定義する。在庫不足が発生した場合のエラーハンドリングを含む。

### 1.2 スコープ

- カートへの商品追加
- カート内容の確認・変更
- 注文情報の入力（配送先・支払い方法）
- 在庫確認・確保処理
- 決済処理
- 注文確定・完了通知
- 在庫不足時のエラーハンドリング

### 1.3 対象読者

- プロダクトマネージャー
- フロントエンド / バックエンドエンジニア
- QAエンジニア

---

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| カート | ユーザーが購入候補として一時的に保持する商品のリスト |
| 在庫確保（仮確保） | 決済処理中に他のユーザーが同商品を購入できないよう一時的に在庫数を減じる操作 |
| 在庫確保解除 | 仮確保した在庫を元に戻す操作 |
| 決済処理 | 外部決済サービスを通じて実際の金銭授受を行う処理 |
| 注文確定 | 決済成功後に注文レコードを確定状態へ遷移させる処理 |

---

## 3. ユーザーストーリー

- ユーザーとして、商品をカートに入れ、スムーズに購入を完了したい
- ユーザーとして、在庫がない場合は購入前に明確に知らされたい
- ユーザーとして、決済中に在庫が切れた場合も、次のアクションが分かるようにしたい
- 管理者として、在庫状況が正確に管理され、過剰販売が起きないようにしたい

## 4. 機能要件

### 4.1 カートへの商品追加

| ID | 要件 |
|----|------|
| CART-001 | ユーザーは商品詳細ページから「カートに追加」ボタンを押すことで商品をカートに追加できる |
| CART-002 | カート追加時に、現在の在庫数を確認し、在庫が0の場合はボタンを無効化し「在庫なし」を表示する |
| CART-003 | 同一商品を複数回追加した場合、カート内の数量を加算する |
| CART-004 | カートに追加できる数量は在庫数を上限とする。上限超過時は上限数量にクランプし、ユーザーに通知する |
| CART-005 | カートの内容はユーザーセッションまたはアカウントに紐づけて永続化する（ログインユーザーはブラウザをまたいで保持） |

### 4.2 カート確認・変更

| ID | 要件 |
|----|------|
| CART-010 | ユーザーはカートページで商品名・単価・数量・小計・合計金額を確認できる |
| CART-011 | ユーザーはカート内の商品数量を変更できる。変更後に合計金額をリアルタイム更新する |
| CART-012 | 数量変更時に在庫数を超える入力がされた場合、在庫上限に制限しエラーメッセージを表示する |
| CART-013 | ユーザーはカートから任意の商品を削除できる |
| CART-014 | カートが空の状態でチェックアウトボタンを押した場合、チェックアウトへ進めない |

### 4.3 チェックアウト開始

| ID | 要件 |
|----|------|
| CHK-001 | ユーザーはカートページから「購入手続きへ進む」ボタンを押してチェックアウトフローを開始できる |
| CHK-002 | チェックアウト開始時に全カート内商品の在庫を再確認する |
| CHK-003 | 在庫不足商品が存在する場合、チェックアウトを開始させず、該当商品と現在の在庫数をユーザーに提示する（詳細は §5 参照） |
| CHK-004 | 未ログインユーザーがチェックアウトを開始した場合、ログインまたはゲストチェックアウトを選択させる |

### 4.4 配送先・支払い方法の入力

| ID | 要件 |
|----|------|
| CHK-010 | ユーザーは配送先住所（氏名・郵便番号・都道府県・市区町村・番地・建物名）を入力できる |
| CHK-011 | ログインユーザーは保存済み住所を選択できる |
| CHK-012 | ユーザーは支払い方法（クレジットカード・コンビニ払い・銀行振込）を選択できる |
| CHK-013 | クレジットカード情報の入力・保持は PCI DSS 準拠の外部決済サービスに委譲する |
| CHK-014 | ユーザーは注文内容確認画面で入力内容を最終確認してから注文を確定できる |

### 4.5 在庫確保（仮確保）

| ID | 要件 |
|----|------|
| INV-001 | 「注文確定」ボタン押下時、決済処理前に全商品の在庫を仮確保する |
| INV-002 | 仮確保はトランザクション内で行い、一部でも在庫不足が発生した場合はすべての仮確保をロールバックする |
| INV-003 | 仮確保の有効期限は15分とする。有効期限内に決済が完了しない場合、自動的に解除する |
| INV-004 | 仮確保中は該当在庫数を他ユーザーから見える在庫数に含めない |

### 4.6 決済処理

| ID | 要件 |
|----|------|
| PAY-001 | 在庫仮確保成功後、外部決済サービスへ決済リクエストを送信する |
| PAY-002 | 決済成功時、注文を確定状態に遷移させ、在庫を正式に減算する |
| PAY-003 | 決済失敗時（残高不足・カード情報エラー等）、在庫仮確保を解除し、ユーザーに失敗理由を表示して支払い方法の変更を促す |
| PAY-004 | 決済リクエストのタイムアウトは30秒とする。タイムアウト発生時は在庫仮確保を解除しエラーを返す |
| PAY-005 | 決済の二重送信を防ぐため、べき等キー（idempotency key）を使用する |

### 4.7 注文確定・完了通知

| ID | 要件 |
|----|------|
| ORD-001 | 注文確定後、ユーザーに注文番号を発行し注文完了画面に表示する |
| ORD-002 | 注文確定後、登録メールアドレスへ注文確認メールを送信する（注文番号・商品一覧・合計金額・配送先・支払い方法を含む） |
| ORD-003 | 注文確定後、カートを空にする |
| ORD-004 | ユーザーはマイページから注文履歴を確認できる |

## 5. 在庫不足エラーハンドリング要件

### 5.1 エラー発生タイミングと対応

| タイミング | 条件 | システムの動作 | ユーザーへの表示 |
|------------|------|----------------|-----------------|
| カート追加時 | 在庫が0 | 追加処理を拒否 | 「この商品は現在在庫がありません」 |
| カート追加時 | 数量 > 在庫数 | 在庫上限数量でカートに追加 | 「在庫数（N個）を上限として追加しました」 |
| カート数量変更時 | 変更後数量 > 在庫数 | 在庫上限にクランプ | 「在庫数（N個）が上限です」 |
| チェックアウト開始時 | 在庫が0になった商品がある | チェックアウト開始を停止 | 「以下の商品が在庫切れになりました：[商品名]。カートから削除するか、数量を変更してください。」 |
| チェックアウト開始時 | カート数量 > 現在庫数 | チェックアウト開始を停止 | 「[商品名] の在庫が N 個になりました。数量を変更してください。」 |
| 在庫仮確保時（注文確定ボタン押下後） | 在庫が不足 | 仮確保全体をロールバック、注文を未確定状態に戻す | 「申し訳ありません。[商品名] の在庫が不足しているため注文を完了できませんでした。カートをご確認ください。」 |
| 仮確保タイムアウト（15分経過） | 決済未完了 | 仮確保を解除、ユーザーに再試行を促す | 「セッションがタイムアウトしました。カートの内容を確認して再度お試しください。」 |

### 5.2 在庫不足時のユーザー導線

- 在庫切れ商品のカートからの削除ボタンをエラー画面上に表示する
- 在庫数が減少した場合は推奨数量（現在庫数）へのワンクリック変更ボタンを提供する
- 在庫切れ商品に代替商品が存在する場合、関連商品リンクを表示する（任意）
- 「在庫入荷通知」機能へのリンクを表示する（任意）

### 5.3 在庫不足エラーのログ・監視

| ID | 要件 |
|----|------|
| ERR-001 | 在庫不足エラーはすべてシステムログに記録する（商品ID・要求数量・実在庫数・タイムスタンプ・ユーザーID） |
| ERR-002 | 在庫仮確保失敗が一定閾値（例：5分間で10件）を超えた場合、運用チームへアラートを送信する |
| ERR-003 | 在庫不足エラー率をダッシュボードでモニタリングできるようにする |

## 6. 非機能要件

### 6.1 性能

| ID | 要件 |
|----|------|
| PERF-001 | カートへの追加・削除操作のレスポンスタイムは95パーセンタイルで1秒以内 |
| PERF-002 | チェックアウトフロー全体（在庫確認〜決済完了）は同時接続1,000ユーザー時に99パーセンタイルで5秒以内 |
| PERF-003 | 在庫仮確保処理は1件あたり500ms以内に完了する |

### 6.2 信頼性・可用性

| ID | 要件 |
|----|------|
| REL-001 | チェックアウト機能の可用性は99.9%以上（月次計算） |
| REL-002 | 決済処理中のシステム障害時、べき等キーにより二重課金が発生しないことを保証する |
| REL-003 | 在庫仮確保と決済処理は結果整合性を確保し、在庫の過剰販売を防ぐ |

### 6.3 セキュリティ

| ID | 要件 |
|----|------|
| SEC-001 | カード情報はシステム内に保持しない（外部決済サービスのトークンのみ保持） |
| SEC-002 | チェックアウト全フローはHTTPS通信を必須とする |
| SEC-003 | 注文・決済APIへのアクセスは認証済みセッションまたはゲストセッショントークンを必須とする |
| SEC-004 | CSRF対策としてチェックアウトフォームにCSRFトークンを実装する |

## 8. 受け入れ基準

| シナリオ | 期待結果 |
|----------|----------|
| 在庫が十分にある商品をカートに追加して購入完了する | 注文完了画面が表示され、確認メールが届く |
| 在庫が0の商品のカートへの追加を試みる | 「在庫なし」が表示され、カートに追加されない |
| カートに10個追加しようとしたが在庫が3個しかない | 3個でカートに追加され、通知メッセージが表示される |
| チェックアウト開始時に在庫が0になっていた | チェックアウトに進めず、在庫切れ商品名と対処方法が表示される |
| 注文確定ボタン押下後に在庫仮確保が失敗した | エラーメッセージが表示され、カートページに戻れる。二重課金は発生しない |
| 仮確保後15分以内に決済を完了しなかった | 仮確保が解除され、再試行を促すメッセージが表示される |
| 決済処理中にネットワークエラーが発生した | 在庫仮確保が解除され、エラーメッセージが表示される。注文は未確定のまま |

---

## 9. 依存・前提条件

- 外部決済サービス（例：Stripe / PAY.JP）との契約・API連携が完了していること
- 在庫管理システムが商品ID単位で在庫数を管理していること
- メール送信サービス（例：SendGrid）が利用可能であること
- ユーザー認証基盤（セッション管理）が整備されていること

---

## 10. 未解決事項・今後の検討課題

| 項目 | 内容 | 優先度 |
|------|------|--------|
| ゲストチェックアウト | 未ログインユーザーの注文情報の保持方法 | 中 |
| コンビニ払い・銀行振込 | 入金確認後の在庫確保タイミング（現状クレジットカードのみ即時確保を想定） | 高 |
| 在庫入荷通知 | 在庫切れ商品の入荷時にユーザーへ通知する機能の詳細仕様 | 低 |
| 複数配送先 | 一注文で複数の配送先に分割する機能 | 低 |
| クーポン・割引 | プロモーションコード適用時の金額計算と在庫への影響 | 中 |
