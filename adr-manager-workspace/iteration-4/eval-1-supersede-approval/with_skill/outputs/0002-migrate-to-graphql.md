## Context

プロジェクトの成長に伴い、複数のフロントエンド（Web、モバイルアプリ、管理画面など）が存在するようになった。
各クライアントが必要とするデータの形状が異なり、REST API ではオーバーフェッチ・アンダーフェッチが頻発している。
また、関連リソースを取得するために複数のエンドポイントを叩く必要があり、クライアント側のクエリロジックが複雑化している。
これらの課題を解決するため、クライアント駆動のクエリが可能な GraphQL への移行を決定した。

## Decision

バックエンド API を REST から GraphQL に移行し、クライアントが必要なデータを単一クエリで柔軟に取得できるアーキテクチャに切り替える。

## Verification (Harness)

```
npm run test:graphql
# スキーマ整合性チェック
npx graphql-inspector validate ./schema.graphql
```

## Consequences

### Benefits（メリット）

- クライアントが必要なフィールドのみを取得でき、オーバーフェッチ・アンダーフェッチを解消できる
- 複数リソースを1回のリクエストで取得できるため、ネットワークラウンドトリップが減少する
- 型付きスキーマにより、フロントエンドとバックエンドの契約が明確になる
- 複数フロントエンドそれぞれの要求に1つの API エンドポイントで対応できる

### Drawbacks（デメリット・トレードオフ）

- チームの GraphQL 習熟コストが発生する
- REST に比べて HTTP キャッシュが使いにくい（クエリ単位のキャッシュ戦略が必要）
- N+1 問題に注意が必要（DataLoader などの対策を要する）
- 既存の REST クライアントとの移行期間中は両方を維持する必要がある

### Side Effects（副作用・注意点）

- REST API（ADR-0001）は段階的に廃止する移行計画を別途策定すること
- API ゲートウェイ・認証レイヤーの設定変更が必要
- フロントエンド側のクライアントライブラリ（Apollo Client 等）の導入が必要
