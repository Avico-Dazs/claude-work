---

## アシスタント

GraphQL への移行は、複数フロントエンドやクエリの複雑化という課題に対して非常に有効な選択です。移行の全体像と手順をご提案します。

### なぜ GraphQL が適しているか（現状の整理）

- **複数フロントエンド** (Web, モバイル等) がそれぞれ異なるデータ形状を必要としている
- REST では over-fetching / under-fetching が発生しやすい
- エンドポイントが増えるにつれてメンテナンスコストが上昇

---

### 移行戦略：段階的移行（Strangler Fig パターン）

一気に切り替えるのではなく、REST と GraphQL を並行稼働させながら段階的に移行する方法を推奨します。

#### フェーズ 1: GraphQL レイヤーの導入（既存 REST を維持しながら）

1. **GraphQL サーバーのセットアップ**
   - バックエンドの言語/フレームワークに合わせてライブラリを選定
     - Node.js: Apollo Server / Pothos / GraphQL Yoga
     - Python: Strawberry / Ariadne
     - Go: gqlgen
     - Ruby: graphql-ruby

2. **スキーマ設計**
   - 既存 REST エンドポイントのリソースをもとに型を定義
   - まずは既存のリソース構造をそのまま GraphQL 型に変換（シンプルに始める）

3. **リゾルバーの実装**
   - 最初は既存の REST API を内部で呼び出す形でリゾルバーを実装（BFF パターン）
   - 段階的に直接 DB アクセスへ移行

#### フェーズ 2: フロントエンドの順次移行

1. **新機能から GraphQL を使用**
   - 既存画面はそのまま REST を使い続ける
   - 新規画面・機能は GraphQL で実装

2. **既存画面の段階的移行**
   - 優先度の高いページや、クエリ複雑性の高い画面から順に移行
   - Apollo Client / urql / TanStack Query (with GraphQL) などを導入

3. **フロントエンド別のスケジュール設定**
   - 各フロントチームと移行タイミングを調整

#### フェーズ 3: REST API の段階的廃止

1. 全フロントエンドが GraphQL に移行完了したエンドポイントから順次 REST を廃止
2. 廃止前に Deprecation ヘッダーや通知を設ける
3. 最終的に REST レイヤーを削除

---

### 重要な考慮事項

#### 認証・認可
- REST で使用していた認証方式（JWT, セッション等）はそのまま流用可能
- GraphQL のフィールドレベルでの認可制御を設計段階で考慮する

#### N+1 問題への対策
- GraphQL は安易に実装すると N+1 クエリが発生しやすい
- **DataLoader** パターンを最初から導入することを強く推奨

#### エラーハンドリング
- REST の HTTP ステータスコードベースのエラーから、GraphQL の `errors` フィールドへの設計変更が必要

#### ファイルアップロード
- GraphQL はファイルアップロードが苦手なため、該当機能は REST のまま残すか multipart を検討

#### パフォーマンス・セキュリティ
- クエリの深さ制限（depth limit）
- クエリの複雑度制限（complexity limit）
- Introspection の本番環境での無効化

### すぐに着手できるアクションリスト

#### 今週中にやること

**1. GraphQL ライブラリの選定・セットアップ**

バックエンドに合わせてライブラリを選んでセットアップします。例えば Node.js + Apollo Server であれば：

```bash
npm install @apollo/server graphql
```

**2. 既存 REST リソースのスキーマ化**

現在の REST API のリソースをリストアップし、GraphQL の型定義（SDL）に変換します。例えば REST に `GET /users/:id` があれば：

```graphql
type User {
  id: ID!
  name: String!
  email: String!
  # ... 既存フィールドをマッピング
}

type Query {
  user(id: ID!): User
  users: [User!]!
}
```

**3. 最初のリゾルバーは既存 REST を呼ぶ形で**

いきなり DB 直接アクセスに変える必要はありません。既存ロジックを再利用しつつ GraphQL 層だけ先に確立します。

```javascript
const resolvers = {
  Query: {
    user: async (_, { id }) => {
      // 既存のサービス層 or REST 呼び出しを再利用
      return userService.findById(id);
    },
  },
};
```

---

#### フロントエンド側の準備

フロントエンドチームと共有すべきこと：

- GraphQL エンドポイント URL（例: `POST /graphql`）
- スキーマの共有方法（Schema Introspection or SDL ファイル）
- 推奨クライアントライブラリ（Apollo Client, urql など）

---

#### DataLoader は最初から入れる

後から追加すると修正箇所が多くなるので、最初から入れておきましょう：

```bash
npm install dataloader
```
