# Conversation: REST API から GraphQL への移行相談

**Date:** 2026-03-22
**Context:** プロジェクトが REST API から GraphQL への移行を検討。複数フロントエンドが存在し、クエリが複雑化してきたことが背景。

---

## ユーザーの質問

> うちのプロジェクト、最初は REST API で設計してたんですが、フロントが複数あってクエリが複雑になってきたので、GraphQL に移行しようと思います。移行手順を相談したいです。

---

## アシスタントの回答

### 基本方針

**段階的移行（Strangler Fig パターン）を推奨。**

REST と GraphQL を一定期間共存させ、リスクを分散する。

---

### Phase 1: GraphQL レイヤーの追加（既存 REST は維持）

1. GraphQL サーバーを追加（Apollo Server、Yoga、Hasura など）
2. 既存 REST エンドポイントをリゾルバでラップする形でスキーマを定義
3. バックエンドロジックは触らず、GraphQL → REST の内部呼び出しで動かす

構成イメージ:
```
フロントエンド → GraphQL API → (内部で) REST API → DB
```

---

### Phase 2: スキーマ設計の精査

- REST の URL 構造をそのままリソースにしない（N+1 問題に注意）
- DataLoader を使ったバッチ取得の設計
- Mutation の設計（REST の POST/PUT/DELETE との対応）
- 認証・認可の GraphQL への移植方法（ディレクティブ or ミドルウェア）

---

### Phase 3: フロントエンドの段階的移行

- 最もクエリが複雑なフロントから移行（GraphQL のメリットが最大化される）
- フロントごとに順番に GraphQL クライアント（Apollo Client、urql など）へ切り替え
- 全フロントが GraphQL に移行完了後、REST レイヤーを廃止

---

### Phase 4: バックエンドの直接接続化

GraphQL リゾルバが REST を内部呼び出しする構成から、DB やサービスに直接接続する構成に整理（パフォーマンス向上）。

---

### 注意すべき課題

| 課題 | 対処法 |
|------|--------|
| N+1 クエリ問題 | DataLoader の導入 |
| スキーマのバージョニング | 非推奨フィールドに `@deprecated` を付けて段階廃止 |
| ファイルアップロード | multipart/form-data の扱いを別途検討 |
| キャッシュ戦略 | REST の HTTP キャッシュとは異なるため、クライアントキャッシュ設計を見直し |
| セキュリティ | クエリの深さ・複雑度制限（depth limit, query cost analysis）を設定 |

---

### 確認事項（ユーザーへの質問）

1. フロントエンドの数と種類は？（Web、モバイル、BFF など）
2. バックエンドの技術スタックは何ですか？（Node.js、Go、Python など）
3. 特に複雑になっているクエリはどんなものですか？（複数リソースの結合、フィールド選択の差異など）
4. 移行のタイムラインや制約はありますか？
