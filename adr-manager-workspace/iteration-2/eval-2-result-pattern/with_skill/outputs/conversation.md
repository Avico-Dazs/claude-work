# 会話ログ — neverthrow 採用決定

**日時:** 2026-03-22

---

## ターン 1

**ユーザー:**
フロントエンドのエラーハンドリングをどうするか悩んでいます。今は try-catch を散在させているんですが、Result 型パターン（neverthrow ライブラリ）を使って統一したいと思っています。どう思いますか？

**Claude:**
良い方向性だと思います。`try-catch` を各所に散在させているケースにおける主な問題点と、neverthrow を採用する際のトレードオフを整理します。

**現状（散在した try-catch）の問題点:**

- エラーの見落としが起きやすい（catch し忘れ、握りつぶし）
- 関数の戻り値の型にエラーが現れないため、呼び出し元がエラーを意識しにくい
- テストで「エラーパスを網羅したか」が不明確になる

**neverthrow の Result 型を使うメリット:**

- `ResultAsync<T, E>` により、成功値とエラーがどちらも型として明示される
- TypeScript の型チェックで「エラーを処理していない」ことがコンパイルエラーになる
- `map` / `andThen` / `orElse` のチェーンで手続き的な try-catch より宣言的に書ける
- テストで「失敗ケース」を自然にカバーできる

**主なトレードオフ:**

- 既存コードのリライトコストが発生する（段階的移行は可能）
- チームが関数型スタイルに慣れていないと学習コストがかかる
- async/await との組み合わせは `ResultAsync` を使う必要があり、多少冗長に見える場面がある

個人的には、フロントエンドでAPIコールやバリデーションが多い場合は neverthrow は非常に相性が良いです。既存コードは境界から少しずつ置き換えていく戦略が現実的です。

採用する方向で進めますか？

## ターン 2

**ユーザー:**
そうですね、それで行きましょう。neverthrow で統一します。

**Claude:**
了解です。Result 型パターン（neverthrow）でフロントエンドのエラーハンドリングを統一する方向で進めましょう。

この決定を検知しました。ADR を記録します。
