# ORM 選定に関する会話ログ

**日付:** 2026-03-22
**テーマ:** Node.js バックエンドプロジェクトの ORM 選定（Prisma vs TypeORM）

---

## ターン 1

**User:**
新しい Node.js バックエンドプロジェクトを始めます。ORM は Prisma と TypeORM で迷っているんですが、どちらがいいですか？型安全性を重視したいです。

**Assistant:**
型安全性を重視するなら、**Prisma** をお勧めします。

主な理由：

**型安全性の面**
- Prisma はスキーマファイル（`schema.prisma`）から TypeScript の型を**自動生成**します。DB のスキーマ変更が即座に型に反映されるため、型のズレによるバグが起きにくい設計です。
- TypeORM も TypeScript 対応ですが、デコレーター + クラスベースのアプローチで、実行時までエラーが見つからないケースがあります。

**開発体験**
- Prisma Client は IDE の補完が非常に強力で、クエリを書くときにリレーションのネストまで型補完されます。
- TypeORM は歴史が長く機能も豊富ですが、設定が複雑になりがちです。

**マイグレーション**
- Prisma Migrate はシンプルで分かりやすく、チームでの管理がしやすいです。

**TypeORM が向いているケース**
- Active Record パターンに慣れている場合
- 既存のデコレーターベースのコードベースがある場合
- より細かい SQL コントロールが必要な場合

型安全性・開発体験・シンプルさを優先するなら、Prisma が現在のエコシステムでは一歩リードしていると思います。

## ターン 2

**User:**
なるほど、Prisma で行きます

**Assistant:**
良い選択だと思います！Prisma を使い始める際のポイントをいくつか：

**セットアップ**
```bash
npm install prisma @prisma/client
npx prisma init
```

**おすすめの進め方**
1. `schema.prisma` でデータモデルを定義
2. `npx prisma migrate dev` でマイグレーション実行
3. 自動生成された Prisma Client を使ってクエリ

**注意点**
- `@prisma/client` は `prisma generate` で型を生成するため、スキーマ変更後は必ず実行を忘れずに
- Prisma Studio（`npx prisma studio`）は DB の内容をブラウザで確認できて便利です

何かセットアップで詰まったことがあれば気軽に聞いてください！

---

## 結果サマリー

- **意思決定:** Prisma を採用することに決定
- **主な決め手:** 型安全性（スキーマから TypeScript 型の自動生成）、IDE 補完の強力さ、シンプルなマイグレーション管理
- **ADR ファイル:** 作成なし（ユーザーから明示的な依頼がなかったため）
