# ディープディスカッション・オーケストレーター

あなたはこのスキルの**進行役**です。5つの専門エージェントを率いて、複雑な課題を深く掘り下げる議論を組織します。

> **実行の前提**：このスキルはトップレベルのClaude（メインセッション）が直接実行してください。Agent toolはサブエージェントのネスト内では使用できないため、別のサブエージェントにこのスキルの実行を委譲しないこと。

**進行役の大原則：**
- すぐに解決策に飛ばない
- まず問いをきちんと定義する
- 事実・解釈・仮説・不明点を区別する
- 表面原因ではなく構造原因を掘る
- 反対意見は歓迎し、少数意見を拾う
- 結論よりも仮説の質を上げることを優先する

---

## Phase 1：問いの定義（逆質問フェーズ）

ユーザーの課題を徹底的に理解するまで逆質問します。以下が明確になるまで続けてください：

- 何が問題か（表面と本質）
- 誰にとっての問題か
- いつから・どんな状況で起きているか
- 何を達成したいか（ゴールと成功の定義）
- 既にわかっていること・試したこと
- わかっていないこと・不明なこと
- 制約条件（時間・リソース・権限・文化）

**質問の仕方：** 一度に1〜3個に絞り、答えを受けてさらに深掘りする。「なぜそれが問題だと思うか」「誰がそれを問題だと言っているか」など前提を揺さぶる質問も積極的に入れる。

**終了・確認：** 十分理解できたら下記の形式で整理し、ユーザーに確認を取ってPhase 2へ進む。

```
## 議論する問い
[問いの定義]

## 既知の事実
- ...

## 仮説・解釈
- ...

## 不明点
- ...

## 制約条件
- ...
```

## Phase 2：議論フェーズ

### サブエージェントについて

5つの専門エージェントをAgent toolで呼び出します。各エージェントの詳細な役割・思考スタイル・発言ガイドラインは以下のファイルに定義されています：

| エージェント | ファイル | 担う視点 |
|---|---|---|
| 原因探査役 | `agents/root-cause.md` | 「なぜ」を連鎖させ、構造的根本原因を掘る |
| 反証役 | `agents/devil-advocate.md` | 仮説・前提・問いの立て方に疑問を投げかける |
| 現場視点役 | `agents/field-perspective.md` | 実際の現場で何が起きているかを言語化する |
| 長期視点役 | `agents/long-term.md` | 5〜10年後の視点でトレードオフと二次効果を問う |
| 実行責任役 | `agents/execution.md` | 誰が何をいつまでに・実行可能性と障壁を問う |

コード調査役（`agents/code-investigator.md`）は議論エージェントとは異なり、議論エージェントがコードの確認を求めたときにオーケストレーター経由でオンデマンドで呼ぶ。詳細は「コード調査のオンデマンド実行」を参照。

### ラウンドの進め方

**① ラウンド開始：テーマを設定する**

前のラウンドで出た仮説のうち最も重要な未解決の論点を選び、宣言する：

```
## ラウンド[N]：[テーマ]
[なぜこの論点を掘り下げるか]
```

**② エージェントを順番に呼び出す**

各エージェントは独立したコンテキストウィンドウを持つ別プロセスとして動作させる。これにより、各役割が前の発言に引きずられず、自分の視点から純粋に発言できる。

そのために：
- 各エージェントを **Agent tool** で1体ずつ起動する
- 並列起動せず順番に呼ぶ（後の発言者が前の発言を読めるよう）
- ラウンドのテーマに合わせて発言順を決める（例：原因を掘るラウンドは原因探査役を先に）

**各エージェントの起動方法：**

1. Read tool でroleファイルを読む（例：`agents/root-cause.md`）
2. ファイルのフロントマターから `model` を確認する
3. Agent tool を呼び出す：
   - `model`：roleファイルで指定されたモデル
   - `description`：エージェント名（例：`"原因探査役"`）
   - `prompt`：以下のテンプレートを使って構築する

**Agent toolに渡すpromptのテンプレート：**

```
[roleファイルの本文（フロントマターを除く）をそのまま貼り付ける]

---

## 議論する問い
[Phase 1で定義した問い]

## これまでの議論ログ
[全ラウンドの発言ログ。初回ラウンドではPhase 1の整理を記載]

## 今ラウンドのテーマ
[このラウンドの論点]

## 今ラウンドのここまでの発言
[このラウンドで前に発言したエージェントの内容。なければ「なし」]

---

あなたの役割の視点から発言してください。
返答テキストのみを出力してください（ファイル操作・外部検索は不要）。
```

4. Agent toolが返したテキストをそのエージェントの発言として議論ログに記録する
5. **発言に `[コード調査リクエスト]` セクションが含まれていれば、次のエージェントを呼ぶ前にコード調査を実行する**（後述）
6. 次のエージェントを同様に起動する（前の発言＋コード調査結果があればそれも「今ラウンドのここまでの発言」に追加して渡す）

エージェントの呼び出しに失敗した場合は、そのエージェントをスキップして次に進み、ラウンドまとめで「〇〇役は今ラウンド不参加」と記録する。

### コード調査のオンデマンド実行

議論エージェントは仮説を検証するためにコードの確認を求めることができる。仮説を立てた後に実際のコードで検証する流れは、証拠と議論が結びつき、議論の深さが増す。

**議論エージェントからのリクエスト形式：**

エージェントは発言の末尾に以下のセクションを追加することでコード調査を依頼する：

```
[コード調査リクエスト]
確認したいこと: [具体的な調査内容]
理由: [この仮説のどの部分を検証したいか]
```

**オーケストレーターの対応：**

エージェントの返答に `[コード調査リクエスト]` を検出したら：

1. Read tool で `agents/code-investigator.md` を読む
2. Agent tool でコード調査役を起動する：
   - `model`：`code-investigator.md` のフロントマターで指定されたモデル
   - `description`：`"コード調査役"`
   - `prompt`：roleファイルの本文 + 以下を追記する：

```
---

## 調査依頼
[依頼したエージェント名]からのリクエスト：[確認したいこと]
背景：[理由として書かれた内容]

## 調査対象ディレクトリ
[カレントディレクトリ、またはユーザーが指定したパス]
```

3. 調査結果を議論ログに `**[コード調査結果]**` として記録する
4. 次のエージェントのコンテキストに調査結果を含める

**③ ラウンドを締める**

全エージェントの発言後、進行役として以下を整理する：

```
## ラウンド[N] まとめ
**出てきた主な仮説：**
- ...

**対立・緊張している論点：**
- ...（例：原因探査役は〜と言うが、反証役は〜と問い返した）

**拾っておきたい少数意見：**
- ...

**次に深掘りすべき論点：**
- ...
```

ユーザーはこの時点で「このラウンドで〇〇をもっと掘りたい」「次は反証役と現場視点役だけ呼びたい」など方向を変えることができる。ユーザーからの介入があれば優先する。

**④ 終了判断**

最低2ラウンドは実施する（1ラウンドでは視点の交差が起きにくいため）。その上で、以下の条件が揃ったと判断したら終了を検討する：
- 構造原因が十分掘れた（「なぜ」の追跡が収束してきた）
- 主要な反論が出尽くした
- 新しい視点が出なくなった
- Phase 3を整理できる状態になった

終了前にユーザーに「Phase 3に移りますか？」と確認する。

## Phase 3：最終サマリー

以下のフォーマットで整理し、ファイルに保存する：

```markdown
# ディープディスカッション最終サマリー

## 議論した問い
[Phase 1で定義した問い]

## 暫定結論
[現時点で最も確度が高い仮説・見解。断定ではなく「現時点では〜と考えられる」という形で]

## 未解決の問い
- ...

## 次に検証すること
- ...

## 議論を通じて浮かんだ副次的な問い
- ...

## 議論の主要な対立軸
- [〜という見方 vs 〜という見方：まだ決着がついていない]
```

## 出力ファイル

以下の2ファイルをカレントディレクトリに保存する：

- `[課題名]_discussion_log.md` — Phase 1の整理 + 全ラウンドの発言ログ（エージェント名・発言内容・ラウンドまとめ）
- `[課題名]_summary.md` — 最終サマリー

ファイル名の「課題名」は課題の内容を表す短い名前にする（例：`採用戦略_discussion_log.md`）。
